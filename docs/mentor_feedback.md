## 2. Ключевой технический фидбек

### 2.1. Нужна мультиагентная система, а не один «суперагент»

Ментор прямо сказал, что **одного агента вам не хватит**:

- Нужен агент, который ищет информацию в интернете;
- Отдельный агент для соцсетей/альтернативных источников;
- Отдельный агент, который **валидирует и агрегирует** результаты;
- Иначе один агент захламит себе контекст и «умрёт» от перегруза.

При этом он не спорит с вашей идеей MCP-слоя (Moex MCP + risk-analytics MCP), но подсвечивает: **над этим слоем** должна работать именно мультиагентка, а не один монолитный интеллект.

> Практический вывод:
> Архитектуру нужно зафиксировать как **multi-agent** (минимум: сбор данных → анализ/оценка риска → валидация/репортинг), а не «один агент делает всё».

---

### 2.2. Агент базы знаний / RAG — под вопросом

Он отдельно спросил: **зачем вам агент базы знаний и как вы его планируете использовать?**

Вы объяснили, что:

- Moex API даёт «сырые» текущие данные;
- Эксперт приносит свои аналитические материалы, модели, заметки;
- Это всё надо подать агенту как knowledge-base (по сути — RAG), чтобы тот понимал «как привычно анализировать рынок» в этой компании.

Фидбек ментора:

- То, что вы описали, **очень похоже на «жирный» системный промпт**, а не на отдельного базу-знаний-агента;
- Он не запрещает вам так делать, но даёт понять, что **в контексте трека это не главное** и может быть переусложнением.

> Практический вывод:
>
> - RAG/knowledge-agent — nice-to-have, но **не должен быть центром архитектуры**.
> - В презентации лучше подать это как «дополнительный слой экспертизы» (тонкая настройка поведения агента), а не как ключевую фичу.
> - На сроки до 12-го его можно вообще вынести за скобки / реализовать минимально.

### 2.3. Финансовый домен = обязательно графики и дашборды

Ментор несколько раз подчеркнул:

- «Если мы говорим про финансы и рынок, **всегда нужны картинки**… Дашборды, графики».
- Эксперты и финдиры живут в дашбордах (SAP, Qlik, BI). Для них текст без визуализации — слабый UX.

То есть **«текстовый консольный агент» для этого кейса — заметный даунгрейд в глазах жюри**. Даже если UI формально не обязателен, визуализация риска портфеля — это почти must-have для вау-эффекта.

---

### 2.4. Agentiq UI / JSON-UI + A2A — сильный буст к оценке

Очень важный кусок:

- Он показывает, что в идеале весь GUI сводится к **одной странице**, а агент **генерирует один JSON**, описывающий интерфейс/виджет.
- Упоминает **Agentiq UI (Agentic UI)** — протокол генеративного интерфейса, который нужно стыковать либо с **ADK**, либо с **Undraft**, и в любом случае с **A2A**, особенно у вас, потому что система сложная.
- Прямой месседж: если вы сможете сделать, чтобы **агент не только считал риск, но и сам генерировал JSON с конфигурацией графиков/дашборда**, который фронт отрисует, — это «точно претендент».

При этом:

- UI не является жёстким требованием («если у бизнеса не обязательно, то нет»),
- Но наличие хотя бы какого-то интерфейса — **заметный плюс в оценке**.

> Практический вывод:
>
> - Базовая линия: сделать **один экран “Risk Cockpit”**, который строится из JSON, сгенерированного агентом.
> - Привязать этот JSON к A2A/AGI UI протоколу, как минимум на уровне DTO.
> - Это прямо попадает в критерий «демонстрация работоспособности + вау-эффект» из общего ТЗ трека.

## 3. Что это всё значит для вашего дизайна и плана

С учётом фидбека ментора и общего ТЗ трека:

1. **Архитектура:**

   - Зафиксировать **multi-agent-подход**:

     - Agent A: сбор и исследование рыночных данных (Moex +, опционально, новости/соцсети) через MCP;
     - Agent B: риск-аналитика и объяснение;
     - Agent C: валидация/контроль качества и генерация финального отчёта/JSON-дашборда.

   - MCP-уровень оставить как сейчас (Moex MCP + Risk MCP), но ясно прописать, какой агент что вызывает.

2. **Knowledge/RAG:**

   - Упростить: описать это как **«доменно-специфичная настройка агента (prompt +, возможно, RAG)»**, без отдельного тяжёлого агента.
   - Реализовать в минимальном варианте, чтобы не похоронить сроки.

3. **UI / визуализация:**

   - Спроектировать **один JSON-контракт для “Risk Dashboard”**, который:

     - содержит конфигурацию графиков / метрик / таблиц;
     - легко мапится на AGI UI / A2A.

   - Сделать простой фронт (или даже ADK-интерфейс), который этот JSON отображает.

4. **Фокус к 12 числу:**

   - Один сквозной сценарий: «загружаем/выбираем портфель → агент планирует исследование → дергает MCP → оценивает риск → возвращает визуальный дашборд + текстовое объяснение».
   - Всё остальное (дополнительные источники данных, полнота мультиагентки, “фancy” RAG) — вторым приоритетом.
